## Why HTTPS Was Added

LAPS was previously served over plain HTTP on port 8080. Every request — including the login form POST that carries the username and password, all session cookies (`JSESSIONID`), and every JWT Bearer token sent to the REST API — travelled across the network in cleartext. Anyone on the same network segment (or at any hop between the client and the EC2 instance) could read or replay these values.

HTTPS encrypts all traffic between the browser and the server using TLS, so an intercepted packet contains only opaque ciphertext. The session cookie, credentials, and JWT are never visible to a passive observer.

---

## Approach: nginx Reverse Proxy (not Spring Boot SSL)

Two approaches exist for adding TLS to a Spring Boot application:

| Approach | How it works | When to use |
|---|---|---|
| **Spring Boot native SSL** | Configure a keystore in `application.properties`; Spring Boot's embedded Tomcat handles TLS | Simple single-server setups; no reverse proxy needed |
| **nginx reverse proxy (chosen)** | nginx listens on 443, terminates TLS, forwards plain HTTP to Spring Boot on localhost:8080 | Production standard; separates TLS concerns from app logic; allows nginx to add security headers and redirect HTTP→HTTPS independently of the app |

nginx was chosen because it separates concerns cleanly: the Spring Boot application does not need to know about TLS at all. nginx is also the industry-standard choice for TLS termination in front of JVM-based applications.

## AWS Academy Constraint: Self-Signed Certificate

Let's Encrypt (the standard free CA) requires a **stable, publicly resolvable domain name** to complete its ACME HTTP-01 challenge. AWS Academy EC2 instances are assigned a new public DNS hostname every time the instance is started:

```
ec2-3-239-228-10.compute-1.amazonaws.com   ← first session
ec2-54-91-12-77.compute-1.amazonaws.com    ← after restart
```

Because the hostname changes between lab sessions, Let's Encrypt cannot issue a CA-signed certificate. The solution is a **self-signed certificate** generated by `openssl` at deploy time using the current `EC2_HOST` value as the `CN` and `subjectAltName`.

**Trade-off:** Browsers display a "Your connection is not private" warning because the certificate is not signed by a trusted CA. This is expected and acceptable — the traffic is still fully encrypted with TLS 1.2/1.3. The warning only means the *identity* of the server cannot be verified by a third party, not that the encryption is absent.

The certificate is **regenerated on every deploy** so it always matches the current EC2 hostname.

## What Was Built — File by File

### 1. `scripts/setup-nginx-https.sh` (new)

An idempotent shell script that provisions nginx and TLS on the EC2 instance. It runs on every deploy via the CD workflow.

**Six steps:**

```
1. dnf install -y nginx openssl          (idempotent — safe to re-run)
2. mkdir -p /opt/laps/ssl && chmod 700   (cert directory, root-only)
3. openssl req -x509 -nodes -days 365    (generate self-signed cert)
   -newkey rsa:2048
   -subj "/CN=${DOMAIN}"
   -addext "subjectAltName=DNS:${DOMAIN}"
   chmod 600 server.key                  (private key root-only, ASVS V6.4.1)
4. Write /etc/nginx/conf.d/laps.conf     (port 80 redirect + port 443 proxy)
5. nginx -t                              (validate config — fail fast before reload)
6. systemctl enable nginx
   systemctl reload|start nginx
```

**nginx port 80 server block (HTTP → HTTPS redirect):**
```nginx
server {
    listen 80;
    server_name <EC2_HOST>;
    return 301 https://$host$request_uri;
}
```

Every HTTP request gets a permanent redirect. The login form is never served over HTTP — credentials cannot be submitted in cleartext.

**nginx port 443 server block (TLS termination + reverse proxy):**
```nginx
server {
    listen 443 ssl;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:...;
    ssl_prefer_server_ciphers off;
    ssl_session_tickets off;

    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Frame-Options SAMEORIGIN always;
    add_header X-Content-Type-Options nosniff always;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-Proto https;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    }
}
```

**Key decisions in the nginx config:**

- `ssl_protocols TLSv1.2 TLSv1.3` — SSLv2, SSLv3, TLS 1.0, TLS 1.1 are all explicitly excluded. These older versions have known cryptographic weaknesses (POODLE, BEAST, etc.).
- **Mozilla Intermediate cipher suite** — a curated list of AEAD cipher suites (AES-GCM, ChaCha20-Poly1305) that provide forward secrecy. No RC4, no 3DES, no export-grade ciphers.
- `ssl_prefer_server_ciphers off` — modern clients negotiate the best cipher they support; forcing server preference is not needed with a well-curated cipher list.
- `ssl_session_tickets off` — session ticket keys are a long-lived secret; disabling them eliminates that attack surface.
- `proxy_pass http://127.0.0.1:8080` — traffic between nginx and Spring Boot is on the loopback interface only. It is not encrypted (unnecessary on localhost) but is not reachable from the network.

### 2. `application.properties` (modified)

Two properties added:

```properties
# HTTPS proxy support (issue #9)
server.forward-headers-strategy=NATIVE
server.servlet.session.cookie.secure=true
```

**`server.forward-headers-strategy=NATIVE`**

When nginx proxies a request to Spring Boot, the original client IP and protocol are lost — from Spring Boot's perspective, every request appears to come from `127.0.0.1` over HTTP. nginx restores this information via headers:

```
X-Forwarded-Proto: https
X-Forwarded-For: <real client IP>
X-Real-IP: <real client IP>
```

`NATIVE` tells Spring Boot's embedded Tomcat to trust and apply these `X-Forwarded-*` headers. Without this setting, Spring Boot would incorrectly treat requests as HTTP, which can break redirect generation and security checks.

**Why `NATIVE` is safe here:** Trusting `X-Forwarded-*` is only safe if port 8080 is not directly reachable from the public internet — otherwise, an attacker could forge these headers. The EC2 Security Group must restrict port 8080 to localhost only (documented in README).

**`server.servlet.session.cookie.secure=true`**

Adds the `Secure` attribute to the `JSESSIONID` cookie. A browser with this attribute set will never send the session cookie over a plain HTTP connection — only over HTTPS. This prevents session hijacking if a user accidentally navigates to the HTTP URL (ASVS V3.4.1).

## How a Request Is Handled — Full Flow

```
Browser (HTTPS)
  → TCP:443 → nginx
      TLS handshake (TLS 1.3, ECDHE key exchange, AES-256-GCM)
      Decrypts HTTP/1.1 request internally
      Adds headers:
        X-Real-IP: <client IP>
        X-Forwarded-Proto: https
        X-Forwarded-For: <client IP>
      → TCP:8080 (loopback) → Spring Boot
          Tomcat reads X-Forwarded-Proto → treats request as HTTPS
          DispatcherServlet → SecurityConfig → Controller → Service → DB
          JSESSIONID cookie has Secure + HttpOnly flags
        ← HTTP response (loopback, internal only)
      Adds headers:
        Strict-Transport-Security: max-age=31536000; includeSubDomains
        X-Frame-Options: SAMEORIGIN
        X-Content-Type-Options: nosniff
  ← TLS-encrypted response

Browser (HTTP)
  → TCP:80 → nginx
  ← 301 Moved Permanently → https://<host><path>
  (login form never loads over HTTP; credentials never sent in cleartext)

Direct access to port 8080 (after Security Group update)
  → EC2 Security Group blocks — connection times out
  (nginx cannot be bypassed from the public internet)
```

## Security Headers Explained

| Header | Value | Purpose | ASVS |
|---|---|---|---|
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | Browser remembers to use HTTPS only for 1 year; prevents protocol downgrade attacks | V9.1.3 |
| `X-Frame-Options` | `SAMEORIGIN` | Prevents clickjacking — page cannot be embedded in an `<iframe>` on another origin | V14.4.3 |
| `X-Content-Type-Options` | `nosniff` | Browser will not MIME-sniff responses; prevents content-type confusion attacks | V14.4.4 |

## Verifying HTTPS Works

After deploy, run these commands to confirm each security control:

```bash
# 1. HTTPS responds with 200 and returns HSTS header
curl -k -I https://<EC2_HOST>/login
# Expect: HTTP/1.1 200 and Strict-Transport-Security header

# 2. HTTP redirects to HTTPS (never serves content)
curl -I http://<EC2_HOST>
# Expect: HTTP/1.1 301 Moved Permanently, Location: https://...

# 3. Session cookie has Secure and HttpOnly flags
curl -k -c /dev/null -I https://<EC2_HOST>/login | grep Set-Cookie
# Expect: Set-Cookie: JSESSIONID=...; Path=/; HttpOnly; SameSite=Lax; Secure

# 4. TLS version and cipher suite
curl -k -v https://<EC2_HOST>/login 2>&1 | grep -E "SSL|TLS|cipher"
# Expect: SSL connection using TLSv1.3 / TLS_AES_256_GCM_SHA384

# 5. Confirm password is NOT in captured network traffic (definitive encryption proof)
sudo tcpdump -i en0 -w /tmp/cap.pcap host <EC2_HOST> and port 443
# (log in via browser, then Ctrl+C)
strings /tmp/cap.pcap | grep -i "password"
# Expect: no output — password bytes are ciphertext, not plaintext
```

## ASVS Controls Implemented

| Control | Where enforced |
|---|---|
| V9.1.1 — TLS 1.2+ only, strong cipher suite | nginx `ssl_protocols TLSv1.2 TLSv1.3;` + Mozilla Intermediate `ssl_ciphers` |
| V9.1.3 — HSTS header present | nginx `add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;` |
| V3.4.1 — Session cookie `Secure` flag | `server.servlet.session.cookie.secure=true` in `application.properties` |
| V6.4.1 — Private key permissions | `chmod 600 /opt/laps/ssl/server.key` in `setup-nginx-https.sh` |
| V13.2.6 — Proxy headers forwarded | nginx `proxy_set_header X-Forwarded-Proto https;` + `server.forward-headers-strategy=NATIVE` |
| V14.4.3 — Clickjacking protection | nginx `add_header X-Frame-Options SAMEORIGIN always;` |
| V14.4.4 — MIME sniffing prevention | nginx `add_header X-Content-Type-Options nosniff always;` |
